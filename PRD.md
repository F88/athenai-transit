# プロダクト要件定義書 (PRD): Athenai

## 1. コンセプト

### 名称の由来

**「Athenai(アテナイ)」**は、「宛てない(目的地を持たない)旅」というコア体験と、古代ギリシャの学問と探索の都「アテナイ」を掛け合わせた名称。

### エレベーターピッチ

Athenaiは、フリー乗車券を持った旅行者や、休日にふらっと街歩きを楽しみたい人のためのWebアプリ。
既存の乗り換え案内アプリが「目的地までの最短ルート」を提示するのに対し、Athenaiは「現在地周辺から次に出発する便」だけを地図とリストで直感的に提示する。ユーザーは画面端のアイコン(エッジマーカー)を頼りに未知のバス停へと歩き出し、直近の出発便を比較検討しながら、思いつくままに次の行き先を決定できる。

### コアバリュー

- **目的地は無い**: あえて目的地入力機能を排除し、「現在地周辺の選択肢」に100%フォーカスする。
- **歩く方向を指し示す「エッジマーカー」**: 現在の地図表示領域外にあるバス停や駅の方向を、画面の端にアイコンとして表示。コンパスのように「あっちへ歩けば乗り物がある」という探索のワクワク感を生み出す。
- **直感的な地図UI(地理院地図 x Leaflet)**: 街歩きに最適な詳細な地形や路地がわかる地理院地図をベースマップに採用し、現在地を中心に周辺の交通機関を可視化。
- **比較検討を楽しむ「直近3便」のボトムシート**: 各路線の次発便は「あと〇分」で強調表示し、次々発以降は絶対時刻(例: 14:30)で表示。飛び乗るか、もう少し散歩を続けるかの意思決定をサポート。

### ユーザー体験のイメージ

例えば、中野の街を散歩していて「次はどこへ行こうか」と思ったとき。

Athenaiを開くと、地図の端にひっそりと小さなバスのアイコンが表示されている。そのマーカーを頼りに路地を歩いていくと、やがて地図上にバス停のピンが現れる。画面下部のリストを引き上げると、「あと5分で出発する、見知らぬ地名行きのバス」があることに気づき、ふらっとそれに乗り込んで新しい街へ向かう。そんな偶然の連続を楽しむことができる。

## 2. 対象ユーザーとデバイス

- **対象ユーザー**: フリー乗車券を持った旅行者、休日にふらっと街歩きを楽しみたい人
- **対象デバイス**: スマートフォン(モバイルブラウザ環境を最優先)
- **バックエンド**: なし。全てのデータ処理はブラウザ内で完結。

## 3. 機能要件 (UI/UX)

### A. 地図表示機能 (メインUI)

- **ベースマップ**: 地理院地図(標準地図)を使用し、街歩きに適した詳細な地形や道を表示する。
- **現在地表示と追従**: Geolocation APIを使用し、ユーザーの現在地を地図上に表示。現在地追従のON/OFF切り替えを可能にする。
- **停留所/駅ピンの表示**: 画面の表示領域(Bounding Box)内にあるGTFSデータの停留所/駅をアイコン(ピン)で表示する。バスと鉄道は視覚的に区別する。

地図レイヤーの z-index 割り当てについては [DEVELOPMENT.md](./DEVELOPMENT.md) を参照。

### B. エッジマーカー (画面外インジケーター) 機能

- 現在の地図表示領域の「外側(例: 画面端から半径1~2km以内)」にある停留所/駅の方向を、画面の四隅や端にアイコンとして張り付けて表示する。
- ユーザーが「この方向に歩けば乗り物がある」と直感できるようにする。
- 地図のスクロール(パン)に連動して、リアルタイムに位置と表示を更新する。

### C. 出発案内リスト (ボトムシート) 機能

- 画面下部に引き上げ可能なリスト(ボトムシート)を配置する。
- 地図の中心、または現在地から最も近い停留所の情報を優先して表示する。
- **表示件数の制限**: 各路線/行き先(系統)ごとに「直近の最大3便」のみを表示する。
- **時刻のUI表現**:
    - 1便目(次発): 現在時刻から計算し、「あと〇分」という相対時間で強調表示する。
    - 2便目/3便目: 「14:30」などの絶対時刻でシンプルに表示し、比較検討の材料とする。
- 経由地などの情報があれば併記し、行き先の想像を膨らませる工夫を入れる。

### D. 選択インタラクション

停留所選択と路線選択は排他的に動作する。同時に両方が選択されることはない。

| 操作                               | 選択状態の変化                | 視覚効果                                       |
| ---------------------------------- | ----------------------------- | ---------------------------------------------- |
| 地図上の停留所をクリック           | stop 選択 ON、route 選択解除  | 該当停留所の路線のみ表示、他の路線は非表示     |
| BottomSheet の停留所をクリック     | stop 選択 ON、route 選択解除  | 地図上の停留所クリックと同じ                   |
| 選択中の停留所を再クリック         | 変化なし                      | -                                              |
| 地図上の路線図(Polyline)をクリック | route 選択 ON、stop 選択解除  | クリックした路線だけハイライト、他の路線は dim |
| 選択中の路線図を再クリック         | 変化なし                      | -                                              |
| 地図背景をクリック                 | stop 選択解除、route 選択解除 | 全路線がデフォルト表示に戻る                   |

選択状態は `selectionInfo`(`routeIds` / `routeType`)を各UIコンポーネントが直接参照して解決する。stop 選択と route 選択は排他的に管理される。

## 4. システムアーキテクチャ

本プロジェクトは、APIサーバーを持たない完全フロントエンド完結のアーキテクチャを採用する。

- **データパイプライン**: Node.js の事前ビルドパイプライン (`pipeline/`) により、GTFS/ODPT オープンデータから最適化済み JSON を生成し、静的ファイルとして配信する。
- **リポジトリパターン**: UI コンポーネントは `TransitRepository` インターフェースにのみ依存し、データソースと疎結合を保つ。

実装の詳細は [CLAUDE.md](./CLAUDE.md) および [DEVELOPMENT.md](./DEVELOPMENT.md) を参照。

## 5. 将来構想

- **GTFS-Flex対応**: 停留所(点)だけでなく、オンデマンド交通の乗降可能エリア(面/ポリゴン)を地図上にオーバーレイ表示し、探索の幅を広げる。
- **PWA化**: オフラインでも次の便を検索できる、完全なお散歩対応アプリへの進化。
- **GTFS-Realtime対応**: リアルタイムの遅延情報を加味した、より正確な出発案内の実現。
